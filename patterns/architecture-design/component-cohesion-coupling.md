# Component Cohesion and Coupling

## Проблема

В большой системе классы группируются в компоненты (jar, dll, gem, package). Без принципов группировки одни компоненты раздуваются, другие зависят от всего подряд, третьи создают циклы. Перекомпиляция и redeploy становятся кошмаром.

## Решение — Cohesion Principles (REP, CCP, CRP)

Три принципа кохезии отвечают на вопрос: «Какие классы объединить в один компонент?»

### REP — Reuse/Release Equivalence Principle

**Гранула повторного использования есть гранула выпуска.**

Компонент, предназначенный для reuse, должен версионироваться и релизиться как единое целое. Пользователь компонента не должен беспокоиться о совместимости отдельных классов внутри — компонент атомарен.

- Классы в компоненте должны образовывать переиспользуемую абстракцию.
- Нельзя взять половину компонента — либо берёшь весь, либо никакой.
- Требует инфраструктуру выпуска (версии, release notes, обратная совместимость).

### CCP — Common Closure Principle

**Классы, которые меняются по одной причине, принадлежат одному компоненту.**

Архитектурный аналог SRP для компонентов. Когда требование меняется, изменения должны затрагивать минимум компонентов — в идеале один.

- Собирай вместе классы, меняющиеся в одном релизном цикле.
- Классы, меняющиеся по разным причинам, разноси по разным компонентам.
- CCP = SRP for components. Следствие OCP: защити компоненты от каскадных изменений.

### CRP — Common Reuse Principle

**Классы, которые переиспользуются вместе, принадлежат одному компоненту.**

Классы редко используются изолированно — обычно группа классов образует переиспользуемую абстракцию. Пользователь компонента не должен зависеть от классов, которые он не использует (→ ISP для компонентов).

- Если ты зависишь от компонента, ты зависишь от **всех** его классов.
- Изменение в неиспользуемом классе компонента вызывает перекомпиляцию/перевыпуск твоего кода.
- Не заставляй пользователей компонента зависеть от того, что им не нужно.

### Треугольник напряжений REP-CCP-CRP

Принципы находятся в напряжении: REP и CCP стягивают классы вместе (большие компоненты), CRP расталкивает (маленькие компоненты).

```
        REP (группируй для переиспользования)
       /  \
      /    \
     /      \
  CCP ----- CRP
(группируй     (разделяй для
 изменения)     пользователей)
```

Архитектор балансирует этот треугольник по мере эволюции проекта: в начале важнее CCP (developability), позже — REP и CRP (reusability).

## Решение — Coupling Principles (ADP, SDP, SAP)

Три принципа связности отвечают на вопрос: «Как компоненты зависят друг от друга?»

### ADP — Acyclic Dependencies Principle

**Граф зависимостей компонентов не должен содержать циклов.**

Цикл означает, что нельзя собрать и протестировать компонент изолированно — он тащит за собой весь цикл. Решение:

- Разорви цикл через DIP: введи интерфейс, на который оба компонента будут зависеть, а не друг на друга.
- Или выдели общую часть в новый компонент.
- Утром понедельника (Monday morning build) проверяй граф зависимостей.

### SDP — Stable Dependencies Principle

**Зависи в направлении стабильности.**

Стабильный компонент тяжело изменить (много входящих зависимостей). Нестабильный — легко (нет входящих зависимостей, много исходящих). Зависимости должны указывать от нестабильных компонентов к стабильным.

```
I (Instability) = Fan-out / (Fan-in + Fan-out)
```

- `I = 0`: максимально стабильный (много входящих, ноль исходящих).
- `I = 1`: максимально нестабильный (ноль входящих, много исходящих).

Бизнес-правила должны быть стабильными (`I ≈ 0`), UI и детали — нестабильными (`I ≈ 1`).

### SAP — Stable Abstractions Principle

**Стабильный компонент должен быть абстрактным.**

Если компонент стабилен (трудно изменить), он должен предоставлять абстракции для расширения. Нестабильный компонент может быть конкретным.

```
A (Abstractness) = Abstract classes / Total classes
```

- `A = 1`: полностью абстрактный.
- `A = 0`: полностью конкретный.

**Зона боли** (`I ≈ 0, A ≈ 0`): стабильный и конкретный — невозможно расширить, невозможно изменить. Пример: БД-схема, от которой зависит вся система.
**Зона бесполезности** (`I ≈ 1, A ≈ 1`): нестабильный и абстрактный — никем не используется, не нужен.

Хороший компонент лежит на **Main Sequence**: `A + I = 1`. Строятся `A/I`-графики для всей системы.

## Tradeoffs

- Метрики `I` и `A` — ориентиры, а не догмы. SDP нарушаемо для стабильных стандартных библиотек (String, `os`).
- Нарушение ADP иногда приемлемо в тесно связанном доменном кластере, но за ним надо следить.
