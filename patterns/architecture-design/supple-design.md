# Supple Design

## Проблема

Доменная модель, правильно отражающая предметную область, может быть громоздкой и неочевидной для разработчика, который пришёл после. Модель требует рефакторинга не только для корректности, но и для читаемости.

## Решение

Supple Design — набор паттернов, делающих модель гибкой и прозрачной. Код должен кричать о том, что он делает, а не шептать.

## Паттерны Supple Design

### Intention-Revealing Interfaces
Имена классов, методов и параметров должны выражать намерение, а не реализацию. `allocateInventory()` вместо `updateStatus(3)`. Читатель кода должен понимать «зачем», не заглядывая в реализацию.

### Side-Effect-Free Functions
Методы [Value Objects](value-objects.md) и [Domain Services](domain-services.md) должны быть чистыми функциями: результат только через возвращаемое значение, без изменения глобального состояния. Упрощает рассуждение о коде и тестирование.

### Assertions
Явные проверки инвариантов на входе в метод и на выходе. Документируют контракт и ловят нарушения на ранней стадии.

### Conceptual Contours
Границы классов и модулей должны соответствовать естественным «изгибам» домена, а не техническому удобству. Если два понятия всегда меняются вместе — они, вероятно, принадлежат одному объекту.

### Standalone Classes
Класс должен быть самодостаточным: максимум смысла при минимуме зависимостей. Легко понять, легко протестировать.

### Closure of Operations
Тип возвращаемого значения операции должен быть того же типа, что и аргументы: `Money.add(Money) -> Money`. Позволяет строить цепочки операций.

### Declarative Design
Описывать свойства и ограничения декларативно (через конфигурацию, атрибуты, DSL), а не императивно.

## Tradeoffs

Плюсы: код — документация, низкий порог входа для новых разработчиков, меньше багов от непонимания модели.
Минусы: требует дисциплины и регулярного рефакторинга, не все паттерны применимы в каждом проекте.
