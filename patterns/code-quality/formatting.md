# Formatting

## Проблема

Несогласованное форматирование увеличивает cognitive load: разработчик тратит ментальные ресурсы на парсинг стиля вместо понимания логики.

## Решение

Профессиональный код выглядит так, будто его писал один человек.

### Вертикальное форматирование

- **Newspaper metaphor**: важное сверху, детали снизу. Имя файла должно отвечать на вопрос «стоит ли читать».
- **Vertical openness**: пустые строки между концепциями. Каждая строка — отдельное выражение. Группа строк — отдельная мысль. Пустая строка разделяет мысли.
- **Vertical density**: строки с тесной связью должны быть рядом, без пустых строк между ними.
- **Vertical distance**: связанные концепции должны быть близко. Protected variables — избегай; переменные объявляй близко к использованию. Instance variables — в начале класса (Java-конвенция). Dependent functions — вызывающий над вызываемым.
- **Conceptual affinity**: функции с семантической близостью (один `assertExpected` перегружен для разных типов) — рядом.
- **Vertical ordering**: high-level → low-level. Caller выше callee.

### Горизонтальное форматирование

- **Line width**: 80–120 символов. Глаз лучше охватывает узкие строки.
- **Horizontal openness**: пробелы вокруг операторов присваивания, между аргументами. Пробел — слабый разделитель, скобки — сильный.
- **Horizontal alignment**: `var1  = 10;` `var10 = 100;` — не нужно. Выглядит как таблица, но маскирует настоящую проблему (слишком много переменных / длинные имена).
- **Indentation**: исходники без индентации нечитаемы. Индентация иерархически отражает scope.

### Team rules

Каждая команда должна иметь единый стиль и **автоматизировать** его проверку. Форматтер + линтер в CI — непременное условие.

## Tradeoffs

- Жёсткий стиль может тормозить в прототипировании. Решается автоматическим форматтером (prettier, gofmt, black).
- Странный legacy-стиль — не менять ради менения, если нет договорённости команды.
