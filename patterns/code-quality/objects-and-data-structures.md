# Objects and Data Structures

## Проблема

Смешение объектов и структур данных приводит к гибридам, которые плохи и как объекты, и как структуры: есть геттеры и сеттеры, но нет настоящего поведения, а get-цепочки нарушают инкапсуляцию.

## Решение

### Data/Object Anti-Symmetry

- **Objects** скрывают данные за абстракцией и предоставляют функции, оперирующие этими данными. Легко добавить новый класс (полиморфизм), но тяжело добавить новое поведение (нужно изменить все подклассы).
- **Data structures** раскрывают данные и не имеют значимого поведения. Легко добавить новую функцию (работает со всеми структурами), но тяжело добавить новый тип данных (нужно изменить все функции).
- **Закон**: процедурный код (data structures + external functions) хорош для добавления функций; объектный код (objects + polymorphic methods) хорош для добавления типов.
- **Гибриды**: half-object/half-data с функциями, делающими значимые вещи, и публичными переменными/геттерами-сеттерами — худшее из двух миров. Избегай.

### The Law of Demeter

Модуль не должен знать о внутреннем устройстве объектов, которыми он манипулирует.

- Метод `f` класса `C` должен вызывать только методы: самого `C`, объекта созданного `f`, переданного как аргумент в `f`, instance variable `C`.
- `ctxt.getOptions().getScratchDir().getAbsolutePath()` — **train wreck**. Нарушает инкапсуляцию (знает внутреннюю структуру `ctxt`), делает код хрупким к изменениям структуры.
- Исключение: цепочки с data structures (DTO, value objects без поведения), если они честно раскрывают структуру.

### Data Transfer Objects (DTO)

- Класс с публичными переменными и без поведения. Полезен для коммуникации с БД, парсинга, сериализации.
- **Active Record** — особый случай: DTO с навигационными методами (save, find). Не добавляй бизнес-логику в Active Record; отдели бизнес-объекты.

## Tradeoffs

- Строгое разделение требует дисциплины и ранних решений о том, где будут данные, а где поведение.
- Train wreck иногда практичен, но каждая точка в цепочке — будущая точка разрыва.
