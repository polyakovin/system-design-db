# Refactoring and Testing

## Проблема

Рефакторинг без тестов — gambling. Без уверенности, что поведение не изменилось, программист избегает рефакторинга, и кодовая база деградирует.

## Self-Testing Code (Fowler)

Fowler вводит концепцию **self-testing code** — код, который сам себя проверяет. Это фундаментальное требование к рефакторингу.

> "Before you start refactoring, make sure you have a solid set of tests. These tests must be self-checking." — Fowler, Chapter 4.

### Характеристики self-testing кода

- **Автоматический запуск:** одна команда → все тесты → pass/fail. Никаких ручных проверок.
- **Самопроверка (self-checking):** тест знает, что такое success и failure, без интерпретации человеком. `assertEquals(expected, actual)` — программа выдаёт pass/fail, а не лог для анализа.
- **Быстрота:** тесты должны проходить за секунды/минуты. Если тесты медленные — их перестают запускать.
- **Изоляция:** падение одного теста не ломает остальные. Тесты независимы.

### Robust test suite as prerequisite

Fowler называет надёжный набор тестов **prerequisite** для рефакторинга. Три причины:

1. **Confidence:** ты знаешь, что не сломал поведение. Каждый шаг рефакторинга верифицируется.
2. **Speed:** автоматические тесты быстрее ручной проверки. Ручная верификация после каждого шага рефакторинга делает процесс нежизнеспособным.
3. **Safety net:** если ошибка всё же проскочила — тесты локализуют её в пределах последнего шага.

### Что делать, если тестов нет?

Fowler признаёт: это парадокс. Чтобы безопасно рефакторить, нужны тесты. Но чтобы написать тесты для legacy-кода, часто нужен рефакторинг.

**Подход:** добавляй тесты постепенно, начиная с самого важного поведения.

1. Напиши characterization tests (Golden Master) — тесты, фиксирующие *текущее* поведение, даже если оно неправильное. Цель: зафиксировать baseline.
2. По мере рефакторинга заменяй characterization tests на спецификационные.
3. Для небольших изменений — используй UI-based approval tests (сравнение скриншотов/вывода).

## Refactoring and Performance

Большинство рефакторингов **не влияет** на производительность измеримо. Fowler разделяет три подхода к перформансу:

### 1. Speculative optimization (неправильный)

«Сделаем быстро сейчас, потому что может понадобиться.» Проблема: premature optimization — ты не знаешь, что будет узким местом. 97% оптимизаций — впустую (Knuth).

### 2. Ignore performance entirely (тоже неправильный)

Программа может стать неприемлемо медленной. Нельзя игнорировать, но и нельзя предсказывать.

### 3. Design for performance later (правильный)

Сначала сделай код чистым через рефакторинг. Затем:

1. Профилируй под реалистичной нагрузкой.
2. Найди узкое место (hot spot) — обычно 5% кода создают 95% latency.
3. Оптимизируй **только** hot spot. Чистый код легче профилировать и тюнить.
4. Перепроверь тесты — оптимизация могла сломать поведение.

> "The secret to fast software, in all but hard real-time contexts, is to write tunable software first, then tune it for sufficient speed." — Fowler

### Когда рефакторинг действительно замедляет

- **Split Loop** — O(n) → O(2n). Почти всегда незначительно. Обратно оптимизировать только если профилировщик показал проблему.
- **Extract Function** — дополнительный вызов в стеке. Современные JIT-компиляторы инлайнят. Не оптимизируй без измерений.
- **Replace Temp with Query** — повторное вычисление. Если дорого — закэшируй.

## When NOT to Refactor

### Большой рефакторинг vs переписывание

Fowler:

- Если код настолько запутан, что рефакторинг займёт больше времени, чем переписывание — **перепиши**.
- Но переписывание дорого: ты теряешь bug fixes и knowledge, накопленные в текущем коде. Часто лучше рефакторить кусками.
- **Strangler Fig** — постепенная замена: новый код пишется с чистого листа и постепенно заменяет старый. Старый код работает, пока новый не готов.

### Слишком близко к дедлайну

- Перед релизом: риск регрессии перевешивает выгоду от чистоты кода.
- «Нет времени» — частый аргумент. Fowler отвечает: рефакторинг — инвестиция. Он экономит время на следующих фичах. Если рефакторинг не делать, каждая новая фича будет занимать всё больше времени.
- Прагматичное правило: не рефактори, если код не будет меняться в обозримом будущем и работает приемлемо.

### Когда код «просто не нравится»

- Рефакторинг ради субъективной эстетики без measurable benefit — wasted effort.
- Критерий: ускорит ли это delivery следующих трёх фич? Если нет — не трогай.

## Tradeoffs

- **100% покрытие не нужно.** Цель — не покрытие, а уверенность. Тесты на критический путь и сложную логику.
- **Slow tests kill refactoring.** Если тесты идут 10 минут — цикл «шаг → тест → коммит» становится 10 минут. Рефакторинг умирает.
- **Characterization tests = худшие тесты.** Они фиксируют текущее поведение, а не правильное. Но они лучше, чем ничего. Постепенно заменяй их.
- **Golden Rule:** Never refactor and add functionality at the same time (two hats metaphor — см. [Refactoring](../fundamentals/refactoring.md)).
