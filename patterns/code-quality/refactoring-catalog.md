# Refactoring Catalog

Полный каталог рефакторингов из Martin Fowler, *Refactoring: Improving the Design of Existing Code* (2nd ed., 2018). Группировка по назначению. Каждый рефакторинг описан по схеме: что делает, когда применять, механизм.

См. также: [Refactoring definition and process](../fundamentals/refactoring.md), [Code smells](code-smells.md).

---

## Fundamental Refactorings

Базовые кирпичики. С них начинается 90% сессий рефакторинга.

### Extract Function

Выделить фрагмент кода в новую функцию и вызвать её по имени.

**Когда:** функция слишком длинная; код требует комментария «что делает»; дублирование.

**Механизм:** выдели код в новый метод → придумай имя, объясняющее *что* делает, а не *как* → проверь, что переменные не выходят за scope → передай локальные переменные как параметры; если изменяются — верни из функции.

### Inline Function

Обратный к Extract Function. Заменить вызов функции её телом.

**Когда:** тело функции так же понятно, как имя; плохая декомпозиция (функция делает слишком мало); цепочка делегирования (indirection без пользы).

### Extract Variable

Ввести поясняющую переменную для сложного выражения.

**Когда:** выражение трудно понять с первого взгляда.

**Механизм:** `return order.quantity * order.price - max(0, order.discount)` → `const basePrice = order.quantity * order.price; const discount = max(0, order.discount); return basePrice - discount`.

### Inline Variable

Обратный к Extract Variable. Убрать переменную и использовать выражение напрямую.

**Когда:** переменная не добавляет ясности; переменная мешает другому рефакторингу.

### Rename Variable / Rename Function

Переименовать переменную/функцию в соответствии с новым пониманием.

**Когда:** имя вводит в заблуждение или не отражает текущее назначение. Самый простой и недооценённый рефакторинг.

### Change Function Declaration

Изменить имя, параметры или возвращаемое значение функции.

**Когда:** сигнатура функции больше не отражает её поведение; параметры передаются неудобно.

**Механизм:** простой случай — просто переименовать; сложный — добавить/убрать/переставить параметры. Миграция: создать новую функцию, вызвать старую как обёртку, постепенно перевести клиентов.

### Introduce Parameter Object

Заменить группу связанных параметров объектом.

**Когда:** Data Clumps — группа параметров всегда передаётся вместе между функциями.

**Механизм:** создать класс, содержащий группу полей → заменить параметры на объект → обновить клиентов.

## Encapsulation

Скрытие деталей и контроль доступа. Фундамент для дальнейшего рефакторинга.

### Encapsulate Record

Заменить свободный доступ к полям записи на геттеры/сеттеры.

**Когда:** данные используются в сыром виде; нужно добавить логику валидации или вычисления.

**Механизм:** создать getter/setter для каждого поля → заменить прямой доступ на вызовы → если record — plain object, создать класс.

### Encapsulate Collection

Контролировать доступ к коллекции — возвращать копию или read-only view.

**Когда:** коллекция доступна напрямую, и клиенты могут мутировать её без ведома владельца.

**Механизм:** добавить `addItem()`/`removeItem()` + getter, возвращающий копию или `unmodifiableList`. Никогда не возвращай ссылку на внутреннюю коллекцию.

### Replace Primitive with Object

Заменить примитивный тип (string, number) на класс с поведением.

**Когда:** Primitive Obsession — примитив обрастает логикой, размазанной по клиентам. Пример: телефон — не просто `string`, а объект с валидацией, форматированием, сравнением.

### Remove Middle Man

Убрать класс-посредник, который только делегирует.

**Когда:** Middle Man не добавляет ценности, а только создаёт indirection. Обратный к Hide Delegate.

### Hide Delegate

Спрятать делегирование — клиент не должен знать о цепочке объектов.

**Когда:** Message Chains (`a.getB().getC().doX()`). Клиент не должен знать, что `a` использует `b`, который использует `c`.

**Механизм:** создать метод в `a`, который делегирует `c.doX()`, и скрыть цепочку.

## Moving Features

Перемещение кода в правильное место. Львиная доля рефакторинга — перемещение.

### Move Function

Переместить функцию в модуль/класс, с данными которого она работает.

**Когда:** Feature Envy; функция чаще обращается к чужому контексту, чем к своему.

**Механизм:** скопировать функцию в целевой модуль → адаптировать → заменить вызовы старого варианта на новый → удалить старый.

### Move Field

Переместить поле в другой класс.

**Когда:** поле чаще используется другим классом, чем своим.

### Move Statements into Function / Move Statements to Callers

**Into:** когда код всегда вызывается перед/после определённой функции — затащи его внутрь.

**To Callers:** обратный — вытолкнуть код из функции в места вызова. Когда нужно различить поведение в разных контекстах вызова.

### Slide Statements

Переместить связанные инструкции ближе друг к другу.

**Когда:** код, работающий с одними данными, разбросан (помогает увидеть Extract Function).

### Split Loop

Разделить цикл, делающий две разные вещи, на два цикла.

**Когда:** один цикл и итерирует, и фильтрует, и накапливает — мешает Extract Function.

**Tradeoff:** O(n) → O(2n), но для читаемости — всегда стоит. Оптимизировать только если профилировщик показал проблему.

### Replace Loop with Pipeline

Заменить `for`/`while` на `map`/`filter`/`reduce`.

**Когда:** цикл реализует map/filter/reduce логику. Pipeline лучше выражает намерение.

### Remove Dead Code

Удалить неиспользуемый код. VCS помнит историю — не храни закомментированный код.

## Organizing Data

Реструктуризация данных для уменьшения связанности и улучшения ясности.

### Split Variable

Разделить переменную, которая используется для нескольких целей.

**Когда:** переменная один раз хранит входное значение, потом — промежуточный результат, потом — аккумулятор.

**Механизм:** завести отдельные `const`/`let` для каждой роли. Идеал — иммутабельные переменные.

### Rename Field

Переименовать поле данных — частный случай Rename Variable для полей структур/классов.

### Replace Derived Variable with Query

Заменить хранимую производную переменную на вычисление.

**Когда:** `total = price * quantity` обновляется при каждом изменении `price` или `quantity` — риск рассинхронизации.

**Механизм:** удалить поле `total`, добавить `getTotal()` → `return price * quantity`.

### Change Reference to Value / Change Value to Reference

**Reference to Value:** заменить изменяемый объект на иммутабельный value object.

**Когда:** объект используется как значение (равенство по содержимому, а не identity). Пример: `DateRange`, `Money`.

**Value to Reference:** обратный. Когда один и тот же логический объект создаётся в разных местах и должен быть одним экземпляром (identity equality). Пример: `Customer`, `Order`.

## Simplifying Conditionals

Условная логика — главный источник сложности. Эти рефакторинги делают её читаемой.

### Decompose Conditional

Выделить условие, then-ветку и else-ветку в отдельные функции.

**Когда:** сложное условие с неочевидной семантикой.

**Механизм:** `if (date.before(SUMMER_START) || date.after(SUMMER_END)) { charge = quantity * winterRate } else { charge = quantity * summerRate }` → `if (isSummer(date)) { charge = summerCharge(quantity) } else { charge = winterCharge(quantity) }`.

### Consolidate Conditional Expression

Объединить несколько условий, ведущих к одному результату.

**Когда:** `if (a) return 0; if (b) return 0; if (c) return 0;` → `if (a || b || c) return 0;`.

### Replace Nested Conditional with Guard Clauses

Заменить вложенные `if/else` на guard clauses с ранним возвратом.

**Когда:** `if (x) { if (y) { return a; } else { return b; } } else { return c; }` — читать тяжело.

**Механизм:** `if (!x) return c; if (!y) return b; return a;`. Ровно одна ответственность на уровень.

### Replace Conditional with Polymorphism

Заменить `switch` по типу на полиморфные классы.

**Когда:** Repeated Switches — одна и та же `switch`-логика в нескольких местах; добавление нового типа требует найти и обновить все `switch`.

**Механизм:** создать базовый класс с абстрактным методом → для каждого типа — подкласс с конкретной реализацией → заменить `switch` на вызов метода.

### Introduce Special Case

Заменить проверки на `null` или специальное значение полиморфным объектом.

**Когда:** код пестрит `if (customer === 'unknown')` или `if (customer == null)`.

**Механизм:** создать `UnknownCustomer` с методами-заглушками (возвращающими дефолтные значения), использовать его вместо `null`.

### Introduce Assertion

Добавить assertion, делающий неявное предположение явным.

**Когда:** код опирается на предположение о состоянии системы, которое не видно из сигнатуры.

**Механизм:** `assert(order.quantity > 0)` — документирует контракт и ловит ошибки на ранней стадии.

## Refactoring APIs

Делаем интерфейсы чище, удобнее и безопаснее.

### Separate Query from Modifier

Разделить функцию, которая и возвращает значение, и производит side effect.

**Когда:** Command-Query Separation (CQS) нарушено. Функция `getTotal()` меняет состояние — это ловушка.

**Механизм:** создать две функции: одна возвращает значение (без side effects), другая — производит изменение.

### Parameterize Function

Объединить несколько функций, отличающихся только константными значениями.

**Когда:** `tenPercentRaise()` и `fivePercentRaise()` — дублирование.

**Механизм:** `raise(factor)` с параметром.

### Remove Flag Argument

Убрать булевый флаг, который меняет поведение функции.

**Когда:** `setDimension(name, value, isOuter)` — флаг говорит «эта функция делает две разные вещи».

**Механизм:** создать две отдельные функции: `setOuterDimension(name, value)` и `setInnerDimension(name, value)`.

### Preserve Whole Object

Передавать целый объект вместо набора его полей.

**Когда:** функция принимает поля объекта как отдельные параметры.

**Механизм:** `calculate(a.low, a.high)` → `calculate(a)`. Если `calculate` не должен зависеть от `a`, создай отдельный параметр.

### Replace Parameter with Query

Заменить параметр вызовом метода, который может получить это значение сам.

**Когда:** вызывающий код вычисляет значение, которое функция может получить сама.

**Tradeoff:** функция становится менее прозрачной (скрытая зависимость). Применять осторожно.

### Replace Query with Parameter

Обратный: передать значение как параметр вместо того, чтобы функция сама его добывала.

**Когда:** нужно сделать функцию чище и независимой от контекста.

### Remove Setting Method

Сделать поле иммутабельным — убрать setter.

**Когда:** поле не должно меняться после создания объекта.

### Replace Constructor with Factory Function

Заменить `new X()` на фабричную функцию.

**Когда:** конструктор делает больше, чем просто инициализацию (валидация, выбор подкласса).

### Replace Function with Command / Replace Command with Function

**To Command:** обернуть функцию в объект-команду, когда нужна сложная параметризация или отмена.

**To Function:** обратный — когда команда избыточна.

## Inheritance Refactorings

Перестройка иерархии классов для правильного распределения ответственности.

### Pull Up Method / Pull Up Field

Переместить метод/поле из подклассов в суперкласс.

**Когда:** одинаковый метод/поле в нескольких подклассах (дублирование).

### Pull Down Method / Pull Down Field

Переместить метод/поле из суперкласса в конкретные подклассы.

**Когда:** метод/поле нужны только части подклассов (Refused Bequest).

### Extract Superclass

Создать новый суперкласс с общей функциональностью.

**Когда:** два класса делают похожие вещи (Alternative Classes with Different Interfaces).

### Collapse Hierarchy

Удалить уровень наследования: объединить суперкласс и подкласс.

**Когда:** суперкласс и подкласс почти не отличаются; подкласс не добавляет существенной ценности.

### Replace Type Code with Subclasses

Заменить поле-дискриминатор типа на полиморфные подклассы.

**Когда:** у класса есть поле `type`, и поведение зависит от него через `switch`.

**Tradeoff:** работает только если тип не меняется после создания объекта (иначе — Replace Type Code with Strategy).

### Replace Subclass with Delegate

Заменить наследование делегированием.

**Когда:** нужно менять «тип» во время жизни объекта; подкласс добавляет только одну вариацию (вторую добавить через наследование уже нельзя); подкласс Refuses Bequest.

**Механизм:** создать отдельный класс делегата → переместить поведение подкласса в делегат → удалить подкласс → клиенты передают нужного делегата.

**Преимущество над наследованием:** композиция гибче. Можно комбинировать делегатов, менять на лету, избегать иерархии ради одного отличия.

### Replace Superclass with Delegate

Если суперкласс не нужен (только часть методов используется, остальное — Refused Bequest), заменить наследование делегированием.

## Tradeoffs

- Не каждый рефакторинг обратим (Extract Function и Inline Function — обратимы; Replace Conditional with Polymorphism — вносит архитектурное решение).
- Механические рефакторинги IDEs делают автоматически (Rename, Extract, Move, Inline) — используй их.
- Порядок важен: сначала Encapsulate, потом Move, потом Extract.
