# Software construction metaphor

## Проблема

Без явной метафоры команда использует неосознанные аналогии, которые формируют процесс и ожидания. Неправильная метафора ведёт к неправильным практикам: например, восприятие софта как строительства здания приводит к жёсткому upfront design и сопротивлению изменениям.

## Решение

McConnell анализирует несколько метафор и их последствия для процесса разработки.

## Метафоры и их следствия

### Строительство здания (building construction)

- **Плюсы:** явные фазы (planning → foundation → structure → finishing), важность upstream design, incremental progress.
- **Минусы:** жёсткая последовательность фаз (waterfall); сложно перестроить фундамент. Софт меняется чаще зданий.
- **Когда полезна:** проекты с фиксированными и понятными требованиями.

### Садоводство / выращивание (gardening / farming)

- **Суть:** софт выращивается, а не строится. Ты планируешь, сажаешь, поливаешь, пропалываешь.
- **Плюсы:** признаёт organic growth и постоянные изменения; поддерживает итеративный подход.
- **Минусы:** можно оправдать отсутствие upfront planning («само вырастет»).
- **Когда полезна:** долгоживущие проекты с evolving requirements.

### Писательство (writing)

- **Суть:** софт — это текст; процесс — draft → review → revise.
- **Плюсы:** признаёт важность ревизий и readability; code review как editing; writing для аудитории (других разработчиков).
- **Минусы:** метафора не охватывает runtime behaviour и системные аспекты.
- **Когда полезна:** подчёркивает важность читаемости кода и iterating on design.

### Аккреция (accretion)

- **Суть:** софт растёт как жемчужина — слой за слоем вокруг ядра.
- **Плюсы:** incremental build-up сложности; реалистично описывает legacy-системы.
- **Минусы:** не даёт guidance для планирования.

### Инженерное проектирование (engineering design)

- **Суть:** софт — инженерная дисциплина с tradeoff analysis, constraints и verification.
- **Плюсы:** closest to reality; признаёт, что «правильного» решения нет — есть tradeoffs.
- **Минусы:** может быть слишком формальным для малых проектов.

## Вывод McConnell

Нет единственной правильной метафоры. Комбинируй: используй engineering design для принятия решений, writing для качества кода, gardening для отношения к изменениям.

## Связь с системным дизайном

Метафора формирует отношение к архитектурным решениям. Команда с метафорой «строительство» будет сопротивляться итеративной архитектуре. Метафора «садоводство» лучше поддерживает evolutionary architecture.
