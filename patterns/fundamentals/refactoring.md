# Refactoring

## Проблема

Код деградирует со временем: требования меняются, hotfix-ы накапливаются, design assumptions устаревают. Без системного рефакторинга код становится неподдерживаемым.

## Определение (Fowler, 2018)

**Рефакторинг** — изменение внутренней структуры программного обеспечения с целью облегчить понимание и удешевить модификацию **без изменения наблюдаемого поведения** (Fowler, Refactoring 2nd ed., p. 44).

McConnell определяет это как **поэтапное улучшение внутренней структуры кода без изменения внешнего поведения**. Это не разовая активность, а непрерывная практика внутри construction-цикла.

Ключевое слово — **observable behavior**. Программа должна работать точно так же до и после. Если поведение изменилось — ты делаешь что-то другое.

### Two hats metaphor (Fowler)

При разработке ты всегда в одной из двух «шляп»:

1. **Adding functionality** — не меняешь существующий код, только добавляешь новое и тесты.
2. **Refactoring** — не добавляешь новую функциональность, не пишешь новые тесты (только если пропустил), только реструктурируешь.

Переключайся между шляпами часто, но всегда осознанно. Никогда не делай то и другое одновременно.

## Процесс рефакторинга (Fowler)

Fowler описывает ритм: **test → small change → test → commit → repeat**.

```
1. Write tests for the area you're about to change (if missing)
2. Make one small refactoring
3. Run tests — all must pass
4. Commit
5. Repeat until done
```

Это TDD-подобный цикл: красный этап пропущен (ты не меняешь поведение), но зелёный обязателен после каждого шага.

### McConnell: процесс рефакторинга

1. **Имей тесты** — перед рефакторингом убедись, что поведение покрыто тестами. Без тестов рефакторинг — это gambling.
2. **Маленькие шаги** — один рефакторинг за раз. Не смешивай рефакторинг с добавлением фич.
3. **Проверяй после каждого шага** — запускай тесты после каждого изменения. Сломал — немедленно откатывай или фикси.
4. **Коммить часто** — каждый успешный шаг в VCS.

### Prerequisites (Fowler)

- **Self-testing code** — без надёжного набора тестов рефакторинг невозможен. Сначала добейся покрытия.
- **Version control** — возможность отката. Git — обязательное условие.
- **Frequent integration** — сливай изменения в mainline как можно чаще, чтобы избежать merge hell.

## Техники рефакторинга

### Основные техники (McConnell)

- **Extract method** — выделение повторяющейся или длинной логики в отдельный метод.
- **Rename** — приведение имён в соответствие с текущим пониманием.
- **Move method/field** — перемещение в класс, где это логически принадлежит.
- **Replace conditional with polymorphism** — замена цепочек if/else на полиморфные вызовы.
- **Introduce parameter object** — группировка связанных параметров в объект.
- **Simplify boolean expressions** — декомпозиция сложных условий на именованные предикаты.
- **Remove dead code** — удаление неиспользуемого кода (VCS помнит историю).

## Рефакторинг и производительность

Большинство рефакторингов **не влияет** на производительность. Fowler выделяет три подхода:

1. **Speculative optimization** (неправильный) — «сделаем это быстрым, потому что может понадобиться». Почти всегда premature.
2. **Ignore performance entirely** — тоже неправильно. Программа может стать неприемлемо медленной.
3. **Design for performance later** — правильный подход. Сначала сделай код чистым через рефакторинг. Затем профилируй, найди горячую точку и оптимизируй **только её**. Чистый код легче профилировать и тюнить.

> "The secret to fast software, in all but hard real-time contexts, is to write tunable software first, then tune it for sufficient speed." — Fowler

## Когда НЕ рефакторить

### Когда проще переписать (Fowler)

- Код настолько запутан, что рефакторинг займёт больше времени, чем переписывание с нуля.
- Код не работает, и ты не можешь понять, как он должен работать.
- Нет тестов, и написать их для существующего кода невозможно без рефакторинга — парадокс: чтобы тестировать, нужно рефакторить, а чтобы рефакторить, нужны тесты.

### Когда слишком близко к дедлайну (McConnell)

- Перед релизом (риск регрессии).
- Без тестового покрытия.
- Когда код работает и не будет меняться в обозримом будущем.
- Когда «просто не нравится стиль» — рефакторинг должен иметь measurable benefit.

### Fowler: ложные причины не рефакторить

- «Нет времени» — рефакторинг экономит время в среднесрочной перспективе. Технический долг растёт экспоненциально.
- «Сломаем» — тесты защищают. Без тестов проблема не в рефакторинге, а в их отсутствии.
- «Не мой код, не трогаю» — коллективная ответственность. Если каждый будет чистить только свой код, система деградирует.

## Связь с системным дизайном

Рефакторинг на уровне кода — аналог evolutionary architecture на уровне системы. Оба признают, что design decisions нужно пересматривать по мере роста понимания. Fowler называет рефакторинг «альтернативой upfront design»: вместо того чтобы пытаться предсказать правильный дизайн заранее, ты развиваешь его итеративно.
