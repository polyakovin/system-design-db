# Feature Flags (Feature Toggles)

Механизм включения/отключения функциональности без деплоя нового кода. Код новой фичи поставляется в production, но скрыт за флагом; флаг переключается через конфигурацию (обычно через dashboard или API). Примеры: LaunchDarkly, Unleash, Split.io, Flagsmith, ConfigCat.

## Где применять

- **Trunk-based development** — мерж фич-веток в main без публикации функциональности.
- **Canary releases** — постепенное включение фичи для % пользователей.
- **A/B тестирование** — показ разных вариантов UI разным сегментам.
- **Kill switch** — аварийное отключение фичи без деплоя при обнаружении бага.
- **Permission gating** — показ фичи только для внутренних пользователей (beta, early access).
- **Operational toggles** — переключение поведения системы (новый алгоритм, другая БД) для rollback.

## Сильные стороны

- **Отделение деплоя от релиза** — код в production, но фича неактивна, пока её не включат.
- **Мгновенный rollback** — выключил флаг → фича пропала; не нужен revert кода.
- **Гранулярный контроль** — можно включать по user ID, региону, случайному проценту, плану подписки.
- **Сокращение merge conflict'ов** — фича-ветки живут недолго, мерж в main раньше.
- **Эксперименты** — объективное измерение влияния фичи на метрики.

## Ограничения

- **Technical debt** — забытые флаги накапливаются в коде; увеличивается сложность тестирования.
- **Усложнение кода** — каждая проверка флага — дополнительный conditional path; читаемость падает.
- **Тестирование комбинаций** — 10 флагов = 2^10 состояний; невозможно протестировать все комбинации.
- **Время жизни флага** — флаг должен жить не дольше, чем фича стабилизируется; stale флаги нужно удалять.
- **Двухфазное удаление** — сначала убрать проверку флага из кода, потом удалить сам флаг из системы.

## Design notes

- **Именование флагов** — `namespace.feature-name.description`, например `checkout.new-payment-flow:on`.
- **Targeting rules** — пользователь → группа → процент → глобально; более специфичное правило побеждает.
- **Client-side vs server-side** — client-side флаги видны в коде клиента (можно обойти); server-side безопаснее.
- **Cache TTL** — кешировать состояние флагов с коротким TTL (30-60s), чтобы изменения применялись быстро.
- **Flag retirement policy** — флаг должен быть удалён из кода max через N недель после стабилизации фичи.
- **Audit log** — каждое изменение флага логируется: кто, когда, что изменил.
